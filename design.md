# 《跳舞吧》iOS MVP 设计方案

## 设计目标

《跳舞吧》是一款面向中老年广场舞人群的极简 iOS 工具型应用。MVP 设计以**一只手可操作、信息大字高对比、关键路径尽量少步骤**为核心原则，采用标准 iPhone 竖屏 9:16 视角组织页面。整体体验参考 iOS 原生应用的层级感与反馈方式，强调大按钮、清晰文案、稳定节奏与明确结果，让用户在较低学习成本下完成“找附近舞团、开始跳舞、加入或创建舞团、导航到场、分享给朋友”等核心动作。

## 关键用户流程

### 1. 首次使用流程

用户首次打开应用后，先看到品牌欢迎页，再进入登录页。完成手机号、微信或抖音任一方式登录后，进入资料完善页。用户设置头像与昵称后即可进入首页。整个流程只保留最少字段，避免复杂注册阻碍。

### 2. 开始跳舞主流程

用户在首页点击巨型“开始跳舞”按钮后，系统先展示定位授权说明并请求定位权限，成功后立即进入拍照流程。用户拍照并确认照片后，系统根据当前位置查询 200 米内的舞团列表。若附近没有舞团，则弹出“是否创建新舞团”的确认；若存在舞团，则展示底部列表供用户选择加入或继续新建。成功创建或加入后，进入成功页，并提供分享卡片与详情页入口。

### 3. 附近舞团浏览流程

用户从首页点击“附近舞团”后进入纯文字列表页，列表按距离从近到远排序，并弱化展示已休眠舞团。用户点击任一舞团后进入详情页，可继续选择导航、分享或联系队长。

### 4. 语音找地流程

用户在首页按住“说话找地”按钮开始录音，松开后转写文本。若文本能模糊匹配到舞团，则进入结果页展示相关舞团；若未匹配到舞团，则展示“直接导航去这个地方”的大按钮。若识别失败，则明确提示“没听清，请再说一次”。

### 5. 队长管理流程

舞团创建者默认成为队长。队长从舞团详情页进入管理页，可查看舞团状态、最近活跃时间，并在舞团休眠时一键唤醒；同时可补充微信群链接，供成员从详情页直接跳转加入。

## 信息架构与导航策略

MVP 采用底部双标签结构与关键动作前置策略。首页作为默认落点，承担最高频动作；第二标签用于集中承载“我的/资料”相关内容，减少层级深度。对用户来说，核心路径尽量控制在两到三次点击内完成。对中老年用户而言，主要操作区位于屏幕中下半部分，避免顶部过高点击负担。

| 一级入口 | 二级页面 | 导航方式 |
|---|---|---|
| 首页 | 开始跳舞流程、附近舞团列表、语音找地 | 推入式导航与底部弹层结合 |
| 舞团详情 | 分享卡片、队长管理、外部地图 | 页面跳转或系统能力唤起 |
| 我的 | 登录状态、资料设置、权限说明 | 标签切换与局部编辑 |

## 视觉与品牌方向

视觉风格应传达“热闹、可信、容易上手”。因此整体采用暖色高对比方案，并辅以少量金色与深灰做层次。主按钮面积大、圆角足、阴影轻，形成类似 iOS 原生主操作按钮的稳定感。文字最小不低于 18，重要信息优先 20 到 32。图标尽量语义直接，避免抽象装饰。

| 设计项 | 方案 |
|---|---|
| 主品牌色 | **#F05A28**，用于主按钮、强调操作、关键状态 |
| 辅助强调色 | **#FF8A3D**，用于渐变、提醒与次主操作 |
| 深色文本 | **#2A241F**，用于标题与主体文字 |
| 浅色背景 | **#FFF7F1**，用于页面底色，降低刺眼感 |
| 卡片底色 | **#FFFFFF**，增强信息块分层 |
| 成功状态 | **#2E9E5B**，用于创建成功、已激活等反馈 |
| 警示状态 | **#C96A1A**，用于权限提醒、休眠提示 |
| 休眠状态 | **#8D8A86**，用于弱化已休眠舞团 |

## 组件与布局规则

首页的“开始跳舞”按钮应占据屏幕宽度约三分之二，垂直位置略低于中心，保证手指自然可达。附近舞团与语音找地入口固定在底部安全区上方，以左右对称布局呈现。列表页使用单列卡片式文本块，每个条目至少 72pt 高度，方便点击。详情页底部固定操作栏包含三个主要操作，按钮宽度等分，文本与图标同时出现，保证识别性。

## 交互反馈原则

所有关键点击都应伴随轻量触感反馈与清晰状态变化。主按钮按下时采用轻微缩放与亮度变化，列表条目按下时降低透明度。流程型页面使用步骤状态提示，例如“正在定位”“正在查找附近舞团”“创建成功”。失败提示使用简短、口语化文案，不展示技术术语。

## 真实数据交互的 MVP 设计边界

本次 MVP 将优先保证以下真实交互能力落地：用户资料保存、舞团创建与加入、附近舞团查询、舞团休眠状态计算、语音文本输入结果承接、外部导航跳转参数生成、分享内容结构化生成。对微信自动建群、微信原生分享卡片、抖音与微信正式 OAuth、运营商一键登录等强依赖第三方审核或正式资质的功能，将采用**可替换接口层 + MVP 降级方案**实现，使整体产品闭环可运行、可演示、可继续接入正式能力。

## 设计结论

《跳舞吧》MVP 的界面设计将围绕“少步骤、大触点、强反馈、真闭环”展开。产品不追求复杂功能堆叠，而是把“开始跳舞”这一核心场景做到足够直接，确保用户能够从首页快速完成定位、拍照、加入或创建舞团，并顺利导航与分享。这一设计方案适合先完成 iOS MVP 验证，再根据真实用户反馈迭代社交与运营能力。

## 本轮 UI 调整说明：向老年友好型极简首页靠拢

基于用户最新提供的参考图，当前《跳舞吧》MVP 的首页需要从“信息卡片型首页”进一步收敛为“单核心动作型首页”。新的首页不再强调数据概览与信息展示，而是强调**看见就会用、点一下就开始**。因此，首页将采用更接近线下老人常见工具型应用的结构：顶部只有品牌标题和极少量功能入口，中部放置一个远大于其他元素的红色主按钮“开始跳舞”，底部只保留两个辅助入口“附近舞团”和“说话找地”，并使用高对比色块与大图标强化识别。

| 调整项 | 新要求 |
|---|---|
| 首页核心结构 | 改为“顶部简洁栏 + 中部单个超大主按钮 + 底部两个辅助按钮 + 底部大标签导航” |
| 主操作优先级 | “开始跳舞”必须成为视觉绝对中心，占据首页最主要面积 |
| 首页信息密度 | 删除不必要的数据卡片、欢迎文案和次级说明，避免老人分心 |
| 颜色策略 | 使用更直接的红、橙、白组合，减少过多层次与浅色信息卡片 |
| 图标策略 | 图标必须大、直观、单义，不使用难理解的抽象视觉元素 |
| 点击热区 | 所有主要按钮必须足够大，适合手指点击，不依赖精细操作 |
| 导航方式 | 底部导航改为更强对比的大标签视觉，类似参考图中的首页/舞队/消息/我的 |

新的首页需要满足以下交互原则。用户打开应用后，不需要阅读太多文字，也不需要理解复杂信息结构，只要看到中央的大红按钮就知道“从这里开始”。附近舞团与语音找地作为次级动作，应在视觉上明显小于主按钮，但仍保持足够大的可点击面积。整体排版应保留充足留白，让每个操作区都能被快速识别。

同时，后续页面也需要跟随首页一起降复杂度。舞团列表、舞团详情、语音找地和个人资料页面都应减少装饰性层次，优先使用单列布局、大字号、短文案和清晰按钮，不再延续偏产品展示型的视觉结构，而改为更接近工具型老年应用的表达方式。
